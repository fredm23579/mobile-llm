# Truthful Build Doctrine

## Core invariant
Every public claim about the project must be backed by executable code, reproducible output, committed tests, or clearly labeled nonfunctional design/spec material.
Unsupported claims are defects.
A README is not marketing copy. It is an interface contract with the user, the developer, GitHub, and the public.

## Non-negotiable rules
### 1. No vaporware claims
Do not write:
> "Supports X"
unless the repository contains working code that actually supports X.
Acceptable wording:
> "Planned: X"
> "Experimental: X, partially implemented in path/to/module"
> "Design target: X, not yet implemented"
> "Currently unsupported: X"

### 2. No fake logs
Do not include logs, benchmark output, screenshots, terminal transcripts, CI results, generated files, demo output, or test summaries unless they were actually produced by the current codebase.
Every log-like artifact should be traceable to:
- command
- commit
- environment
- date
- output file

### 3. No README inflation
The README may describe only:
- what exists
- how to run it
- what it actually does
- what it does not do
- known limitations
- how to reproduce the demonstrated output

It must not imply that unbuilt features are already available.

### 4. No burden shifting
The user should not have to discover missing dependencies, hidden setup assumptions, undocumented env vars, broken commands, unavailable files, non-existent APIs, or imaginary examples.
- If setup requires something, document it.
- If a command is listed, it must run.
- If a dependency is required, it must be declared.
- If a feature is incomplete, say so before the user wastes time.

### 5. No deception by omission
A claim can be false even when every sentence is technically defensible.
**Bad:**
> "Run the pipeline with `python main.py`."
*(when the pipeline only works after manually creating folders, downloading private files, editing hardcoded paths, and setting undocumented environment variables.)*

**Truthful:**
> "`python main.py` currently runs only after configuring DATA_DIR and providing input files in ./data/raw. No downloader is included yet."

## Required release gate
Before anything is pushed publicly, the repo must pass this checklist.

# Release Truthfulness Checklist

## Build and run
- [ ] Fresh clone works.
- [ ] Dependencies install from declared files only.
- [ ] Main advertised command runs.
- [ ] README quickstart has been tested exactly as written.
- [ ] No undocumented manual steps are required.
- [ ] No hardcoded private/local paths remain.
- [ ] No missing files are referenced.
- [ ] No private credentials, tokens, or secrets are required for public demo mode.

## Documentation honesty
- [ ] README claims match implemented functionality.
- [ ] All incomplete features are labeled incomplete, planned, or experimental.
- [ ] No future-tense roadmap item is described as current capability.
- [ ] No benchmark/result/log/demo is included unless generated by the repo.
- [ ] Limitations are stated plainly.
- [ ] Known failure modes are listed.
- [ ] Installation instructions were tested from a clean environment.

## Evidence
- [ ] Test command included.
- [ ] Example command included.
- [ ] Example output can be regenerated.
- [ ] Generated artifacts are either committed intentionally or excluded intentionally.
- [ ] CI result, local test result, or reproducibility note exists.

## Public integrity
- [ ] No fake stars, fake usage claims, fake adoption claims, or fake testimonials.
- [ ] No claims of production readiness unless deployment/reliability work actually exists.
- [ ] No claims of security, privacy, correctness, compliance, or robustness without supporting implementation.
- [ ] No misleading comparison to other tools.
- [ ] No hidden dependency on unavailable external services.

## README language standards
**Use this**
> ## Current status
> This project currently implements:
> - Local file ingestion from `./data/input`
> - Basic parsing for `.txt` and `.md`
> - JSON export to `./output/result.json`
> - A CLI entrypoint: `python -m project_name`
> 
> The following are not implemented yet:
> - PDF parsing
> - Web crawling
> - Authentication
> - Cloud deployment

**Avoid this**
> A powerful AI research platform that ingests documents, crawls the web, analyzes everything, generates reports, and supports enterprise workflows.
*(unless the code physically does all of that.)*

## Reproducible demo standard
A legitimate demo should be runnable from a clean checkout.
```bash
git clone <repo-url>
cd <repo>
make
./mobile_llm --chat
```
The README should then show output like:
> Expected output:
> [Interactive Chat Mode Started. Type 'exit' to quit.]
> User> 

**Not:**
> Amazing output generated!
*(unless that exact line is produced by the program.)*

## Minimal repo structure for honesty
The repository must be cleanly organized and formally gated by a CI workflow that proves the README’s minimum promise. `mobile_llm` implements this via `.github/workflows/cmake.yml`.

## Strong rule for AI-generated repos
AI-generated code must be treated as untrusted until proven executable.
Do not publish AI-generated README claims until after:
1. clean install
2. actual run
3. actual test
4. actual output inspection
5. README correction
6. limitation disclosure

## The brutal operational standard
A repo is not ready because the idea is good.
A repo is ready when a stranger can clone it, follow the instructions, run the advertised command, reproduce the advertised result, and not discover that the README was borrowing credibility from code that does not exist.
